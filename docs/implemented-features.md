# 已实现功能说明

本文档记录当前版本已经落地的能力，作为后续验收、交接和继续迭代的基线。

## 项目形态

- 单机版 Web 应用，基于 Next.js App Router、React、TypeScript 实现。
- 数据库使用 SQLite，本地文件持久化，默认路径可通过 `DATABASE_PATH` 配置。
- UI 使用 Arco Design。
- 支持 Docker Compose 部署，包含 Web 服务和同步 worker。
- 运行配置支持 `.env` 覆盖，包括访问密码、Session 密钥、数据库路径、飞书 API 地址、同步轮询间隔和时区。
- 后台页面访问需要输入统一访问密码。
- `APP_SECRET` 按需求明文入库，不做加密。

## 当前代码架构

当前代码采用按业务功能组织的模块化单体结构，保留现有业务能力和 HTTP 契约，不保留旧的层级式兼容目录。

- `src/app`：Next.js App Router 适配层。`(dashboard)` 路由组统一处理页面鉴权和应用壳，API 路由只负责 HTTP 输入输出。
- `src/modules/auth`：密码登录、签名 Cookie、鉴权和登录/退出界面。
- `src/modules/companies`：公司公开契约、服务端仓储、编辑、排序和手动同步入口。
- `src/modules/schedules`：crontab 领域规则、计划仓储、应用服务和管理界面。
- `src/modules/organization`：组织领域模型、差异计算、每日快照、部门树、用户/部门/历史仓储和界面。
- `src/modules/overview`：公司概览和近期入离职查询与界面。
- `src/modules/sync`：飞书传输、组织读取、原始响应采集、同步编排、事务持久化和 worker 调度。
- `src/shared`：数据库连接与 schema、运行配置、日志、HTTP 参数、分页、时间和共享 UI。
- `tests/modules`：按业务模块组织的单元与集成测试；`tests/contracts` 保存数据库和依赖边界契约。

依赖方向固定为 `app/worker -> modules -> shared`。领域目录不依赖 Next.js、SQLite 或 UI 框架，共享层不反向依赖业务模块。详细说明见 `docs/architecture.md`。

## 导航与页面

当前左侧导航顺序为：

1. 概览
2. 员工列表
3. 部门树
4. 公司主体

桌面端使用固定左侧导航；宽度不超过 `720px` 时切换为固定底部导航，页面内容、工具栏、表格、弹窗和抽屉会按手机宽度自适应。

页面整体去掉了说明型副标题，保留更清爽的业务操作界面。

## 概览

概览页按公司主体展示卡片，一个公司一张卡片。

每张卡片展示：

- 公司名称
- 人员数量：当前在职员工数
- 历史总数量：累计员工变更事件数
- 近 7 天入职人数：近 7 天内新增或恢复在职的员工数
- 近 7 天离职人数：近 7 天内标记离职的员工数

概览卡片顺序跟随公司主体中维护的公司排序。

## 公司主体

公司主体页面已支持：

- 新增公司主体
- 编辑公司名称、`APP_ID`、`APP_SECRET`、启用状态
- 查看公司最近一次同步状态
- 查看最近同步的部门数、用户数、新增数、更新数
- 同步失败时展示错误信息
- 手动触发公司同步
- 打开并维护公司 crontab 定时任务
- 拖动公司行进行排序

公司排序会作为全局公司顺序使用，影响：

- 公司主体列表
- 概览页公司卡片顺序
- 员工列表的公司下拉顺序和默认公司
- 部门树的公司下拉顺序和默认公司
- 其他基于公司列表的展示入口

排序字段持久化在 SQLite。当前应用不执行数据库结构迁移，部署时数据库必须符合当前 schema。

## 定时任务

同步定时任务采用 crontab 表达式。

已支持：

- 一个公司主体配置多条 crontab
- 新增 crontab
- 编辑 crontab
- 删除 crontab
- 启用或停用单条 crontab
- 查看上次触发时间
- 查看下次触发时间
- worker 按轮询间隔扫描到期任务并触发同步
- 同一公司有多条到期 crontab 时，会合并为一次同步执行，避免重复拉取
- 公司仍支持手动触发同步

crontab 格式为 5 段：分钟、小时、日、月、周，例如 `0 9 * * *`。

## 飞书数据同步

同步流程已实现：

- 获取 tenant access token
- 拉取部门列表
- 拉取员工列表
- 归一化飞书部门和员工字段
- 写入当前部门表、当前员工表、员工部门关系表
- 记录同步运行状态、开始时间、结束时间、统计信息和错误信息
- 同步期间的错误会写入同步记录，并在公司主体页面展示

用户接口分页已按飞书接口方式处理，支持持续拉取多页员工数据。

## 员工当前数据

员工当前信息保存在 `users_current`。

当前展示和查询维度包括：

- 头像
- 姓名
- 邮箱
- 岗位
- 部门
- 上级
- 状态：在职、离职
- 最近出现时间

员工列表页面已支持：

- 按公司查询
- 状态筛选：全部、在职、离职
- 默认展示在职
- 按姓名、邮箱、岗位搜索
- 查看单个员工变更历史
- 展示头像，头像缺失时使用姓名首字兜底

## 用户每日快照

用户快照已落库到 `user_daily_snapshots`。

当前规则：

- 以公司、用户、日期为唯一维度
- 每个用户每天只保留一份快照
- 同一天多次同步时会覆盖当天快照
- 快照日期按应用配置时区计算，默认 `Asia/Shanghai`
- 快照保存同步时的完整归一化用户数据和常用冗余字段

该设计用于满足“用户信息维度每天只存一份”的要求。

## 飞书原始同步快照

每次完整拉取部门和员工接口后，会在 `sync_raw_snapshots` 中新增一条不可覆盖的原始快照，并通过 `sync_run_id` 与同步任务一一关联。

快照的 `payload_json` 保存：

- 部门递归查询的每一次请求及每一页飞书响应
- 按部门查询员工的每一次请求及每一页飞书响应
- 当前业务模型未使用的飞书扩展字段
- 原始分页字段和请求 URL

认证接口响应、tenant access token、公司 `APP_ID` 和 `APP_SECRET` 不写入原始快照。只有部门和员工数据全部拉取成功后才创建快照；如果后续归一化数据入库失败，原始快照仍会保留并关联失败的同步任务。

原始快照采用追加写入，不按天覆盖。它包含员工联系方式等敏感数据，并会随同步次数持续增加数据库体积，部署时需要同步考虑数据库文件的访问权限、备份和保留策略。

## 变更历史

员工变更事件保存在 `user_change_events`。

已记录的事件类型：

- `created`：员工首次出现
- `updated`：员工字段发生变化
- `resigned`：员工在同步结果中消失，被标记为离职
- `restored`：已离职员工重新出现在同步结果中，恢复在职

每条事件包含：

- 公司 ID
- 员工 open_id
- 事件类型
- 变更字段列表
- 变更前快照
- 变更后快照
- 发生时间 `occurred_at`
- 关联同步任务 ID

变更历史展示已优化：

- 按天分组展示
- 同一天内多条变更按时间倒序展示
- 每次更新可展示多个字段
- 展示字段名、变更前值、变更后值
- `extra` 扩展字段会拆成具体子字段，例如 `extra.description`
- 支持对状态、部门、岗位、邮箱、上级、头像等字段做历史展示

## 部门树

部门树页面已支持：

- 按公司展示部门树
- 点击部门查看员工
- `直属员工` tab：仅展示当前部门直属员工
- `下属员工` tab：展示当前部门所有下级部门员工，不包含当前部门直属员工
- 表格字段包括用户头像、用户名、岗位、部门、上级、操作
- 点击员工详情按钮打开弹窗
- 弹窗展示员工当前信息和变更历史

## 部门取消处理

部门同步已从硬删除优化为软删除。

当前规则：

- 新同步中仍存在的部门标记为 `active`
- 之前存在但本次同步不存在的部门标记为 `deleted`
- 记录取消时间 `deleted_at`
- 员工仍引用已取消部门时，界面展示部门名并标注“已取消”
- 默认部门树只展示 active 部门

这样可以避免部门被取消后，历史员工或离职员工的部门信息直接丢失。

## 数据表概览

核心表包括：

- `companies`：公司主体、飞书凭证、启用状态、排序
- `company_sync_schedules`：公司 crontab 定时任务
- `sync_runs`：同步运行记录
- `sync_raw_snapshots`：每次同步获取到的飞书原始分页响应
- `departments`：部门当前状态，含软删除字段
- `users_current`：员工当前状态
- `user_departments`：员工和部门关系
- `user_daily_snapshots`：员工每日快照
- `user_change_events`：员工变更事件

当前 schema 已删除旧版固定间隔同步字段，例如 `sync_enabled`、`sync_interval_minutes`、`last_sync_at`、`next_sync_at`。同步时间以 `sync_runs` 和 `company_sync_schedules` 为准。
